Re: does a sparse NVECTOR implementation make sense?

From Radu Serban <serban1@llnl.gov>

Andrey Romanenko wrote: 
Hello,

Will it make any sense to try to implement a sparse version of the NVECTOR module? I have looked at the Meschach library and it seems that it should be doable to write up the stubs. Don't know if this can be potentially useful in the SUNDIALS solvers, though. Thank you.

Regards,
Andrey Romanenko


Andrey,

I assume you mean a sparse linear solver, not a sparse vector module... Yes, a sparse linear solver does make a lot of sense and Fortran predecessors of the SUNDIALS solvers (GEARS and LSODES) have such an option. As an example, such a solver would be very useful for large-scale problems for which a preconditioner is hard to construct (e.g. chemical kinetics problems).

The challenge however is in designing a sparse linear solver module which would work with a parallel NVECTOR implementation. The difficulty is not the linear solver per-se (there are several which we could interface to) but rather the generation of a sparse Jacobian using difference-quotient approximations. As you have probably noticed, none of the linear solver modules in SUNDIALS require the user to provide a Jacobian function; they all provide the default option of estimating the Jacobian matrix or matrix-vector products by finite differences. For a large, sparse Jacobian, one needs to find sets of variables that can be perturbed simultaneously for the FD Jacobian approximation. This can be posed as a graph coloring problem and doing this in a distributed parallel environment is not a resolved problem.

The good news is, I am currently working on exactly this problem. We've made some progress and, hopefully, SUNDIALS will provide a sparse linear solver module soon. However, if you are only interested in serial computations, it should be no problem interfacing any sparse solver to one of the SUNDIALS solvers, especially if you can also provide a Jacobian function.
