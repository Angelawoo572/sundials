> > Argh, I obviously overlooked CVodeSetErrFile... I, however, 
> just had a 
> > quick look back at the source code and noticed that there is a fair 
> > number of calls to "fprintf" with "stderr", so that 
> wouldn't solve my 
> > problem. The problem is that my application is a pure 
> Windows one and 
> > not a console application, so there is no easy way for me to allow 
> > COVDE to output errors to "stderr" or elsewhere.
> If CVodeSetErrFile is called as soon as possible (i.e. right 
> after CVodeCreate) then the only error messages that are sent 
> to stderr are those from CVodeCreate (indicating an illegal 
> input or a memory allocation failure) and those indicating 
> that the cvode memory is NULL (i.e. a CVODE function is 
> called before CVodeCreate).
> 
> (Note: in the current release there are a few places where 
> fprintf unnecessarily uses stderr, but these have been fixed 
> for the next release).
> 
> My point is that such errors are quite unlikely with a 
> minimum of carefulness and so all meaningful error messages 
> can actually be redirected or disabled with CVodeSetErrFile.

Ok, I didn't want to go into computational details, but the fact is that my
application is written in Borland Delphi (object oriented pascal) and that I
compile the CVODE library using Borland C++ compiler to create .obj files
that I then link to my Delphi project. In doing that, I had to "rewrite" a
few functions that don't "directly" exist in Delphi, namely: "malloc",
"free", "pow", "sqrt" and "fabs".

Now, if I want to allow for all those "fprintf" statements, no matter
whether to "stderr" or to a file, I will have to add something for streams
and "fprintf" (if I recall correctly). Those ones are a bit more tedious to
"rewrite" and I must confess I don't have time to do it, hence my suggesting
using compiler directive... :)

> Adding a compiler directive to suppress output could lead to 
> problems for users that use the SUNDIALS libraries installed 
> by someone else.

I think you will end up with problems of some sort or another, no matter
what you decide. I agree, though, that one should minimize the number of
potential problems and if that means not having a compiler directive, then
so be it. I will then just keep doing what I am doing I guess, unless
someone has a better idea as how to embed a C library to a Delphi project
(i.e. *NO* DLL).

> > One could, in fact, have a function that would return the error 
> > message (based on the error code), if required.
> I can see at least two reasons why this is not advisable:
> 1) To keep the number of error codes to a minimum, the same 
> one may be used for different situations (e.g. CV_ILL_INPUT) 
> which are differentiated only through the error message.

Why would that be a good reason? Personally, I would never do that: each
error should, in my view, have an error code associated to it.

> 2) Some error messages also print various quantities (the 
> current time, step size) which are context dependent and 
> would be difficult to pass to such a function.

There is always a way... :)

> > Personally, I would very much welcome such a function, as I 
> had to go 
> > through all the important bits of the library to extract the error 
> > messages that could potentially help me debug my 
> application, should 
> > CVODE fail to integrate one of my systems of ODEs.
> All CVODE constants (including error codes) are listed in 
> Chapter 9 of the user guide.

Thanks for that. That reminds me that there are lots of error codes that are
used for different errors. Take "-1" for instance, it's used for
"CV_MEM_NULL", "CV_DENSE_MEM_NULL", "CV_BAND_MEM_NULL", "CV_DIAG_MEM_NULL",
etc. Again, I would, personally, avoid such a thing.

        Alan.

PS: don't get me wrong, what you guys have achieved with the CVODE library
is great! My users and I definitely enjoy the computational speed up that
comes from using CVODE.
