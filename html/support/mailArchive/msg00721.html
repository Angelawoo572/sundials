Yes and in fact I have the felling it is even worse than that, since it
would seem that it needs to allocate 3 times that amount (for M, savedJ and
pivots through respective calls to DenseAllocMat, DenseAllocMat and
DenseAllocPiv).

Having no idea about the numerical algorithms behind CVODE, I guess I was
wondering whether it was normal to request so much memory. I guess the
answer is yes, which unfortunately doesn't help me, since the problem I
mentioned earlier (i.e. ~580 virtual cells) is very small indeed (my
problems can easily require 2 millions virtual cells).

> But, and that may be the problem, you request these 424 Mb in 
> a continuous memory array, which the operating system may 
> have a hard time finding. If you debug to the line 41 in smalldense.c
> 
> a[0] = (realtype *) malloc(n * n * sizeof(realtype));
> 
> you can check here the requested amount of memory.

Well, for the above problem, it requires ~1.5 GB of RAM, which I normally
have (I have 2 GB), but maybe not at the time I was doing my tests.

> Solutions:
> 
> - compile sundials with realtype set to float, you loose 
> accuracy, but also reduce the required amount of dense matrix 
> storage by a factor of 4

True, but that would still not solve my problem (see above).

> - buy more RAM? May work, but I'm not sure... Theoretically 
> there shouldn't be a limit on how big the continuous memory 
> block can be that you request with malloc.

I run my problems on a desktop computer, so 2 GB is as much as I can get on
a 32-bit system.

> - implement your own dense-matrix package where data storage 
> is organized such that each row has its own memory block 
> instead of allocating all at once,
>    may result in page swapping and be slower, but it would 
> allow for clustered memory

> - use an iterative solver which does not require inversion of a matrix

I think any of those solutions would involve quite a bit of work and, to be
honest, I would rather try to stick with what the Sundials library offers
me, if possible.

Hmm... It would seem that for the time being I will have to stick to my
current solution...
