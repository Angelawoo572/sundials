Andreas,

I'm not quite sure I understand what it is you describe here. Maybe you can be a bit more clear (for example, what is the context of the 7199.99 value you mention?).

In any case, I think I should again point out that the "t" in the call to CVode below is just an OUTPUT value. It has no influence whatsoever on the step size (as it seems to me you think/suggest). When the integrator approaches t_stop (which, in your case, is equal to t_out) it will not take progressively smaller steps (but rather only adjust the last step size that would take it beyond t_stop), so I don't see how you can possibly "underrun" the minimum step size (at least not for this reason; maybe there is something else going on?)

The first "Usage Notes" on the updated SUNDIALS website (accessible from the Support page) discusses in more detail the differences between t_out, t_ret, t_stop, and the internal steps (t_n) taken by CVODE. The discussion there may be helpful.

--Radu

P.S. Note that the call to CVodeSetStopTime makes no sense inside the inner loop. You should call it only once after set_next_t_out.

Andreas Nicolai wrote: 
Hi there,

I just tripped over a small problem with CVODE, consider the following code which uses CVODE in one step mode:

------------------------
while (t < t_end) {
   set_next_t_out(t_out);

   while (t < t_out) {
       CVodeSetStopTime(cvode_mem, t_out);
       result = CVode(cvode_mem, t_out, y_storage, &t, CV_ONE_STEP_TSTOP);
       switch (result) {
            // ERROR handling
       }
   }
}
------------------------

While this looks completely reasonable, it has a small flaw in it :-)

When at some output time point the boundary conditions change and the solver is forced to reduce the time step to a very small value, I get several calls to CVODE with the same t_out but t's that continuously approach my t_out until my minimum time step is underrun.

Essentially the server returns a successful completion, but t = 7199.9992. Which is, of course, the correct - yet unexpected - outcome of the call. After all the documentation states that the value of t is only meaningful in case of an error, otherwise we have reached the t_out successfully. The code above works fine when I add the line:

if (results >= 0)
   t = t_out;

Maybe this could be added as some kind of notice/warning in the documentation?

Bye,
Andreas
