While I agree with almost everything that has been posted in this thread, I'd like to point out that many (probably most) of our users do not perform system-wide installations of SUNDIALS, as they do not have write permissions to /usr, /usr/local, /opt, etc. As such, I do not favor exporting all SUNDIALS header files under a 'sundials' subdirectory of $includedir. A user installing SUNDIALS with --prefix=/home/user/my_sundials for example would end up with a directory /home/user/my_sundials/include which would only contain a subdirectory 'sundials' (and further /home/user/my_sundials/include/sundials/sundials).

Moreover, any of the installation layouts described by Andre can be obtained by 
specifying --includedir and --libdir instead of --prefix.
For example: ./configure --includedir=/usr/local/include/sundials 
--libdir=/usr/local/lib

Is this too much to ask from those that want system-wide installs?

--Radu

Andreas Nicolai wrote:

Hi,

ok, seems we have a conflict of interests here :-) Essentially there are
two requirements to be met:

Windows - Header and Lib files under the same base/install directory

...

Considering that under Windows a "make install" is typically not done or
even necessary,
...



As developer and administrator you have to to differentiate between
build location (with all source files and a lot of temporary files
created while compilation) and the clean install directory (specified
by prefix) in the file system.

It is a very bad practice NOT to use "make install" and IMHO
it has nothing to do with Windows or UNIX/Linux.


Under Windows an installation of the library is typically just "unzip in
some user defined dir, open the project for your IDE, compile and link
your projects to whatever dir you have everything in".



I have never seen a typically layout under Windows.
If there is any philosophy it will be changed with the
next Windows version.  ;-)

As I see the proposal by Andrey provides a solution
for two scenarios:

a) global installation of SUNDIALS in default
   directories:

   prefix:  /usr

   /usr/include/sundials
   /usr/include/sundials/ida.h
   /usr/include/sundials/ida/ida_band.h
   ...
   /usr/lib/libsundials*.a
   /usr/lib/libsundials*.so   (sundials*.dll)

   or for shared file systems (e.g. NFS):

   prefix:  /usr/local

   /usr/local/include/sundials
   /usr/local/include/sundials/ida.h
   /usr/local/include/sundials/ida/ida_band.h
   ...
   /usr/local/lib/libsundials*.a
   /usr/local/lib/libsundials*.so   (sundials*.dll)

b) separate installation for providing different
   SUNDIALS versions in parallel

   prefix:  /opt/sundials-2.2.0

   /opt/sundials-2.2.0/include/sundials
   /opt/sundials-2.2.0/include/sundials/ida.h
   /opt/sundials-2.2.0/include/sundials/ida/ida_band.h
   ...
   /opt/sundials-2.2.0/lib/libsundials*.a
   /opt/sundials-2.2.0/lib/libsundials*.so   (sundials*.dll)

   or for shared file systems (e.g. NFS):

   prefix:  /usr/local/sundials-2.2.0

   /usr/local/sundials-2.2.0/include/sundials
   /usr/local/sundials-2.2.0/include/sundials/ida.h
   /usr/local/sundials-2.2.0/include/sundials/ida/ida_band.h
   ...
   /usr/local/sundials-2.2.0/lib/libsundials*.a
   /usr/local/sundials-2.2.0/lib/libsundials*.so   (sundials*.dll)

For Windows maybe "C:\Programs\SUNDIALS" will be used as prefix.

   C:\Programs\SUNDIALS\include\sundials\ida.h
   C:\Programs\SUNDIALS\include\sundials\ida\ida_band.h
   ...
   C:\Programs\SUNDIALS\lib\sundials*.dll

Of course, in all SUNDIALS source files the appropriate
path should be used for #include

   #include "sundials/ida.h"
   #include "sundials/ida/ida_band.h"
   ...

IMHO only this would be a clean approach - in many
other software projects exactly the same solution
are used.
