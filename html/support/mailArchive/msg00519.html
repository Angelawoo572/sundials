Hi Alan G,

I have used CVode on a similar problem like yours. Here are the few
implementations I tested:

1) Operator Splitting with one CVode object
This is the easiest to set up and memory requirements for CVode
are neglibile compared to the state vectors for the individual cells.
This is the slowest method (as you have also observed) beacuase
we need to reinitialize CVode with different cell's states
for every time step.


2) ODE+PDE coupled 
In this implementation, I coupled the ODEs and PDEs and used
a large CVode object to solve the resulting ODEs. In this case
we don't need to reinitialize CVode for every timestep.
This method was the fastest as long as number of cells remained
small (in my case around 64 cells). As soon as the number
of cells increased beyond that threshold, the performance was significantly
reduced. I don't know why.

3) Operator splitting with seperate CVode objects for
 every cell
This approach is reasonably fast (at least in my experience so far) for any
number of cells as long as the memory requirements are within
the system's range.
I am able to solve 160,000 cells simultaneously on 2.8GHz pentium
4 machine with 4GB memory. For more cells, I have to reinitialize
my CVOde objects (similar to the first approach).
I didn't have to modify any of the CVode parameters.

> Hi Alan,
>
> I asked for the equations because I thought you might be
> solving some DAEs with an operator split approach.
> But for your system, you should certainly rethink how you use
> CVODE. A fully implicit approach will give you much better
> performance than the operator splitting you are currently
> doing. Just semi-discretize your PDE with any finite
> difference and use CVODE on the resulting
> (large) ODE system. Several of the CVODE examples (all of
> those using one of the Krylov linear solvers but also cvbanx)
> are good starting templates.
>
> You will have to pay attention to all the issues Alan H.
> mentioned (ordering of variables, choice of linear solver,
> and, if using one of the Krylov solvers, preconditioning).
>
> I would worry about memory requirements only after getting
> CVODE to work on the proper ODE system. In any case,
> commenting out portions of the code will not save you much
> (if anything) and will only make updating to newer SUNDIALS
> releases harder. If you do not use root finding, there is no
> extra memory allocated, so it's useless to comment it out.
> Also, the memory allocated for stability limit detection is
> insignificant (an array of 24 realtypes). If memory really
> becomes an issue, you can save much more by limiting the
> maximum method order (but only after careful consideration of
> the pros and cons) since the space required for N_Vectors
> dominates (by a large extent) the CVODE memory usage.
>
> --Radu

> > At the moment, the way I solve that is:
