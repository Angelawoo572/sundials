Hello Andreas,

I agree with most of your advice about this problem.  However,
there are a couple of points that I'd like to clarify:

(1) In general, I would expect that the RHS function f depends
continuously on y, even if y is subjected to limits.  That is,
I expect f is continuous w.r.t. y_local.  And since y_local is a
continuous function of y, f is still continuous w.r.t. y itself.
Thus the solver should not have too much trouble in this case.
This use of y_local is the correct way to handle problems with such
constraints.

(2) You say there may be cases where for small y_i > 0, f_i is much
different from 0, while for y_i <= 0, f_i = 0.  In that case, the model
itself is discontinuous, and should be handled differently, as follows:
Assuming y_i starts out positive, first define a continuous form of
the model, in which f_i does not jump when y_i reaches 0.  Then solve
the IVP with rootfinding turned on, the root function being y_i, so
that you stop the problem when y_i = 0.  Then flip a switch that sets
f_i = 0, and restart the IVP solution.

-Alan Hindmarsh



On Tue, 24 Jan 2006, Andreas Nicolai wrote:

> Hi there,
>
> I got a question about CVODE that I might better answer in the user group. It 
> might be of general interest.
>
> > I have essentially 3 state variables here. If I don't' do
> > DummyY[i]=Y[i]<0? 0:Y[i] then my code runs in 10 secs (with errors, -ve
> > values etc.), When I restrict the slow variables to +ve, it still runs ok
> > but when I restrict the fast variable it takes 12 Hrs.
>
> This refers to range restriction of variables. Often physical 
> quantities/variables are limited to a certain typical range, for instance 
> concentrations/masses to positive numbers or in my case a moisture content 
> must not exceed a maximum moisture content.
>
> The straightforward solution, that was proposed in one of the last posts, is 
> to restrict the values obtained from the solver like:
>
> y_local[i] = std::max(0.0, y[i]);
>
> or in a different fashion for an upper and lower limit
>
> y_local[i] = y[i] < 0.0 ? 0.0 : ( y[i] > y_max ? y_max : y[i] );
>
> However, that will/might result in a discontinuity of your calculated 
> derivative in y_dot. For non-stiff systems that is typically not a problem 
> because the variables will change _slowly_ and the solver can quickly find 
> the iterative solution. However, for stiff systems sometimes the range around 
> the limitation gives the solver a hard time to converge. This becomes 
> particularly nasty if the difference quotient calculation of the Jacobian is 
> affected.
>
> Essentially the gradient of y[i] "jumps" at each limit. For example:
>
> dy (y > 0) = some value different (sometimes much different) from 0
>
> but
>
> dy (y <= 0) = 0
>
> The solution that I use to avoid that "jump" is to ensure a smooth transition 
> in the derivative of y. At the limit, say y[i] = 0 the derivative should be 
> 0, slightly away (say a delta_y away) from that limit the derivative should 
> be the actual derivative that is calculated. In between the derivative (or, 
> if it directly depends on the y value, also the variable itself) should be 
> "smoothed" out so that you have a smooth transition from a certain number to 
> zero.
>
> In my code I multiply my y values with a scaling function, that is applied 
> when the y values are "close" to a limit. So the actual code I use in my f() 
> function looks like:
>
> for (unsigned int i=0; i<n; ++i)
>      y_local[i] = limit(y[i]);
>
> and the funtion limit() looks like (pseudo code, because in my case it is 
> much more complicated):
>
> double limit(double y) {
>     // LOWER_LIM = DELTA_Y
>     // UPPER_LIM = Y_MAX - DELTA_Y
>     if (y > LOWER_LIM) {
>        if (y < UPPER_LIM)   return y;
>        // scale between UPPER_LIM and Y_MAX
>        double delta = Y_MAX - y;
>        // use parabolic function or any other function type approximate
>        // gradients at UPPER_LIM and Y_MAX (at Y_MAX, delta = 0 and dy should 
> be 0)
>        // determine SCALE_UPPER based on a) try-and-error or b) based on your 
> equations
>        return delta < 0.0 ? Y_MAX : Y_MAX - SCALE_UPPER*delta*delta;
>     }
>     else {
>        // scale between 0 and LOWER_LIM
>        return y[i] < 0.0 ? 0.0 : SCALE_LOWER*y[i]*y[i];
>     }
> }
>
> The actual scaling function is parabolic: SCALE*delta*delta. In my code I 
> actually use a cosine scaling function and an exponential function, because 
> they simply resulted in better convergence. So trying out some other 
> functions is certainly recommended.
