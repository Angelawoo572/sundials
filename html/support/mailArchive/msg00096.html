Herbert Sauro wrote:

I'm tempted to use swig to wrap sundials/IDA, but it looks as if the
internal format for the numerically intensive operations in IDA will be
NVector. (I'm OK restricting myself to serial mode because the system
size is small).

Until now, I've been using the Numeric package for Python that has its
own array storage conventions and means for accessing, but I'm concerned
that NVector types would require an effort to adapt to Numeric array
types.




As one of the SUNDIALS development team, I'm also a Python user. I've started a background (aka evening/weekend) project of a Python implementation of parts of SUNDIALS. At this point, I've done the generic NVector routines and implemented and tested about 2/3 of the routines for a scalar NVector implementation. The operations table is simply a Python directory with the keys being the generic function name and the values being references to the implementation functions. Each NVector carries a reference to the operations table, which is implemented within the module namespace. The underlying data arrays are handled via Numeric Python arrays, methods and ufuncs. Coding this does take a little care in notation. Just about everything in Python is a reference to an object. In coding the NVector operations, it's important that the data within the referenced object be changed rather than the reference being reset to point to a new object.

My intent is to implement the shared linear solver routines after I've finished the NVector routines.

As a native Python implementation, use should be straightforward. Doing a wrapper, would require getting the right Python NVector routines called. The C routines in SUNDIALS wouldn't actually care about the data format, as long as the correct operations are performed and the right norms returned.

Anyway, that's a summary of something in progress to acknowledge the growing Python popularity and create an active role for SUNDIALS within that.
