Re: [sundials-users] include file directory layout

From "John W. Eaton" <jwe@bevo.che.wisc.edu>
Date Mon, 19 Jun 2006 17:18:46 -0400 
On 19-Jun-2006, Radu Serban wrote:

| While I agree with almost everything that has been posted in this thread, I'd 
like to point out that many (probably most) of our users do 
| not perform system-wide installations of SUNDIALS, as they do not have write 
permissions to /usr, /usr/local, /opt, etc. As such, I do not 
| favor exporting all SUNDIALS header files under a 'sundials' subdirectory of 
$includedir. A user installing SUNDIALS with 
| --prefix=/home/user/my_sundials for example would end up with a directory 
/home/user/my_sundials/include which would only contain a 
| subdirectory 'sundials' (and further 
/home/user/my_sundials/include/sundials/sundials).

I don't see that this really matters.  If a user can't install in
/usr or /usr/local or /opt, it seems natural to me to think of
whatever they choose for $prefix to contain a filesystem structure
that would be exactly the same as one would find if $prefix could be
/usr or /usr/local or opt.  Instead of using
$prefix=/home/user/my_sundials for SUNDIALS and /home/user/my_emacs or
whatever, I would recommend using /home/user/all-my-local-software and
installing all the packages I need there, not just SUNDIALS.  Then the
extra level in the directory structure doesn't seem so odd.

But if it bothers you, then why not just put all the include files in
one directory, say $prefix/include/sundials (in the above, $prefix
would be /home/user/my_sundials?  It seems to me that there are not so
many components of SUNDIALS that this would be a bad solution.  Then
users of SUNDIALS would put

  #include <sundials/whatever.h>

in their code and specify -I$prefix/include in the compiler command
line flags.  Isn't that the simplest solution for the users?

| Moreover, any of the installation layouts described by Andre can be obtained 
by specifying --includedir and --libdir instead of --prefix.
| For example: ./configure --includedir=/usr/local/include/sundials 
--libdir=/usr/local/lib
| 
| Is this too much to ask from those that want system-wide installs?

Whatever you choose, it would be best if all the software written to
use Sundials can be system independent with respect to the way the
include files are found.  Please don't make the installation different
on different types of systems.

I think it would be nice if we could write

  #include <sundials/whatever.h>

and have this work on all systems, also preferably without having to
specify more than one -I flag on the command line.

Sure, individuals can modify the location of the installed files at
configure time or just by copying files to some directory structure
they like.  But if everyone uses different layouts, it makes it
difficult for us to write portable software that uses SUNDIALS.  I
don't want to have to use different #include statements depending on
whether I'm building for Windows, Unix, OS X, OS/2, or whatever.

jwe
