Hi all,

I completely agree with John.

John W. Eaton wrote:
> On 19-Jun-2006, Radu Serban wrote:
> 
> | While I agree with almost everything that has been posted in this thread, 
> I'd like to point out that many (probably most) of our users do 
> | not perform system-wide installations of SUNDIALS, as they do not have 
> write permissions to /usr, /usr/local, /opt, etc. As such, I do not 
> | favor exporting all SUNDIALS header files under a 'sundials' subdirectory 
> of $includedir. A user installing SUNDIALS with 
> | --prefix=/home/user/my_sundials for example would end up with a directory 
> /home/user/my_sundials/include which would only contain a 
> | subdirectory 'sundials' (and further 
> /home/user/my_sundials/include/sundials/sundials).
> 
> I don't see that this really matters.  If a user can't install in
> /usr or /usr/local or /opt, it seems natural to me to think of
> whatever they choose for $prefix to contain a filesystem structure
> that would be exactly the same as one would find if $prefix could be
> /usr or /usr/local or opt.  
>
> ...
> 
> 
> I think it would be nice if we could write
> 
>   #include <sundials/whatever.h>
> 
> and have this work on all systems, also preferably without having to
> specify more than one -I flag on the command line.
          ^^^^^^^^^^^^^^^^^^^^^
This is the most important point for me. We want to include the
SUNDIALS package in our simulation environment MOSILAB. All subcomponents
use $MOSILAB/include and $MOSILAB/lib for headers and libraries.
Only for the SUNDIALS package we have to add two (or more) additional
include paths ($MOSILAB/include/sundials, $MOSILAB/include/ida, ...)
since version 2.2.0.

Second reason is the administrator's point of view ...


/André
