Andreas Nicolai wrote:
Am Wed, 25 Jan 2006 12:33:00 -0500 hat Radu Serban <serban1@llnl.gov> geschrieben:

Without having really thought it through, I see a couple of issues (I'll use CVODE as an example):

1) This would still leave out the "memory NULL" error. In each function that receives as first
argument the solver (or preconditioner) memory pointer an error is returned if this pointer is NULL
(by the way, this is one of the reasons why it's always good practice to initialize such pointers to
NULL before calling a function like CVodeCreate). But then we couldn't access the error handling
function stored in cvodemem...

I think in this case falling back on stderr would be feasible. The
calling (windows/UI) program could check the pretty specific return code in this case.

All sundials solvers and sub-modules (linear solver, preconditioner, etc) return -1 (under different names) so indeed this would be easy to capture in the calling program. But this would impose a constraint on any developer of a new module.

2) Theoretically, each module (e.g. linear solver) should have its own error handling function
(since we allow for users to define new such modules). But that is probably overkill, so we need to
think of a mechanism that would allow using the same function.

That leads either to a global pointer, that is commonly used by modules if not NULL or a pointer argument that is passed on to all modules from the main solver module (would require changing lots of interfaces). A global pointer to the _current_ error handler function seems to be the best solution (even though global==evil, but then again, stderr is global, too).

A possible way to do that would be to set the error handler via a global SundialsSetErrorHandler() function and subsequent called modules would use that error handler function. This would ensure that different modules all use the same error handler (if NULL they use stderr).

Actually, everything is in place to use the same error handler in all modules since a pointer to the main solver memory is passed to all modules. What I was getting at was not access to this function from a sub-module, but rather the issue of uniquely identifying an error (and also passing module-specific information to the error handler).

So maybe the function type for such an error handling function should also contain an argument
(const char*) identifying the module?
void error_handler(int error_code, const char *module, const char *msg, void *data);

This sounds good. I wouldn't expect to have too many different modules in future so that it becomes impossible to manage unique identification names.

No, there shouldn't be an unmanageable number of modules. An other option would be to impose that a module appends its name to the error message before sending it to the error handler, but I think I prefer the extra argument.

Just my 2 cents,
Andreas
