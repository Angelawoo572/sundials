Dear Hannes, 
 
I just finished setting up an Matlab / IDA solution. Its not a general
wrapper but a specific 
solution to my problem and it was pretty easy to do. I'm not using mex
but rather the matlab compiler
which is a bit different but if you want to use mex it shouldn't be hard
as well. 
Here is what I did:

1. wrote the C part and made sure it works as C "main" function. 

2. wrote a mex gateway function, there are several example in the matlab
doc as to how to do it. 
It basically  parse the input and output and call your 'real' other
function. look for matlab example timestwo.c
and others like that for reference. 

3. translated the main from step 1 to be the 'real' function the
mexFunction gateways calls. 

4. compile. This was a bit tricky. I couldn't figure out a way to link
the IDA libraries using mcc
so I first only generated the c code using mcc -c... then build it using
mbuild. I think if you use 
mex and not mcc it should be straight forward regular
-I/usr/.../sundials  and -lsundials_ida etc. 

I guess you could use these steps to build a general interface but
writing the function that calls 
the solver in C and calling that function through the gateway
mexFunction was pretty simple as well. 

Its really worth while doing, using IDA was 50(!) times then matlab's
ode15s, at least in my case. 

Hope this helps, 

Roy

APPENDIX 1: My background story and motivation:

I have been solving sets of DAEs (up to 117 highly coupled and complex
DAEs) with MATLAB's DAE solver ode23t.m but encounter numerical problems
as for example high computational effort (for example 28min computing
time for 10 sec of simulation time in the case of 75 DAEs) and
non-converging integration processes possibly due to ill-conditioned
Jacobian matrices. These DAEs were symbolically generated by heavily
using the Symbolic Math Toolbox, converted to a so-called inline
function and an anonymous function, and then fed to the solver. All this
is done in MATLAB

Earlier trials brought that I am able to compile DASSL and an according
mex interface for MATLAB by Steinar Hauan
(http://www.andrew.cmu.edu/user/steinhau/Research/mex-interfaces.html)
but running it would result in segmentation violation errors. This has
been tried using MATLAB (7.1 and 6.5.1), Cygwin/mingw/gnumex, and later
also with mingw/msys without any success. I have to stick to
windows/MATLAB/Real-Time-Workshop since the drivers for my (controls)
experiment are distributed for windows/MATLAB/Real-Time-Workshop, only.
I also thought about working on a linux system using WINE but I am not
sure if WINE would be capable of interfacing MATLAB/Real-Time-Workshop
and the drivers for the controls hardware correctly.

Since DASSL won't run on my computers I have been trying to make other
DAE solvers written in C or fortran work so for example IDA (DASKP
written in C) of SUNDIALS.

I am aware of the mex interface between DAE solver RADAU and MATLAB
written by Christian Ludwig
(http://www-m3.ma.tum.de/twiki/bin/view/Software/ODEWebHome and
http://www.unige.ch/~hairer/software.html
<http://www.unige.ch/%7Ehairer/software.html> ) and currently try to
make it work, too.

I am also aware of the SLICOT libraries
(http://www.slicot.de/start.php?site=archives) but have not seriously
touched them, yet.
