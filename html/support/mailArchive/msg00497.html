I don't know where you guys got the idea that in a windows dev environment all 
files in a project have to be in the same directory. Of course they don't. 
Today I compiled CVODE 2.4 using VS 7., not a hitch. And I didn't have to move 
a single file.

I do not believe we need a different layout for Windows, the current one works 
fine. All I do is run configure using cygwin then tell VS 7 where the include 
and C source folders are and hit the compile button (Non-Windows OSs do the 
same thing but using makefiles). I don't know what the user population of 
sundials is with regards to OS usage, but depending on demand it would be nice 
to have some VS projects files with the distribution for those who are not 
familiar with the ./configure approach to installing source code.

Herbert Sauro

On 13-Jun-2006, Radu Serban wrote:

| We decided to export header files in subdirectories in order to not pollute 
the include/ directory (especially for a system-wide SUNDIALS 
| install). I would have gladly use "#include <ida/whatever.h> where necessary 
(I like *NIX too :-)) but we had to make it relatively easy for 
| those using other OS (read Windows) who cannot use the configure script and 
thus need to collect all source and header files into a 
| directory where their IDEs (e.g. Visual C++) can find them.

Is it really so difficult for them to install things in multiple
directories when using Visual C++?

| With the current implementation they do not have to create these 
subdirectories.
| 
| I know this is a bit more inconvenient, but there are not that many -I flags 
that you need to add to CPPFLAGS (typically 3: -I***/include 
| -I***/include/ida -I***/include/sundials).

I think it would be much better to be consistent on all systems, and
to do things in a way that would not require adding a bunch of -I
flags to the compiler command line.  But if you really must have
different directory layouts on different systems, then how about at
least giving us the option of installing things in multiple
directories, and not having to specify multiple -I options on the
command line?  If you did something like

  #if defined (INCLUDES_IN_ONE_DIRECTORY)
  #include <whatever.h>
  #else
  #include <ida/whatever.h>
  #endif

in your sources, then I think we could just use

  #include <ida/ida.h>

in our builds and not have to add any extra -I flags to the compiler
command line.

Or, instead of all that mess, how about using a single subdirectory
called sundials for all these files, and using

  #include <sundials/whatever.h>

in your sources?  That would avoid polluting the /usr/include
directory, make it easy for VC++ users who want things in one
directory, and allow the install to be the same on all systems.

jwe
