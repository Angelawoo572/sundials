From "John W. Eaton" <jwe@bevo.che.wisc.edu>

On 20-Jun-2006, Radu Serban wrote:

| I also looked at my installation of octave (I have an older version, 2.1.65) 
and its header files are installed under a directory 
| ${prefix}/include/octave-2.1.65/octave. This means that even if 
${prefix}/include is in the compiler's default search path, an example using 
| #include <octave/whatever.h> still needs a -I flag. John, correct me if this 
is not true.

Yes, the situation with Octave is not ideal as it requires adding both
-I$prefix/include/octave-VERSION and -I$prefix/include/octave-VERSION/octave
to the compiler command.  We should probably fix this so that at most
one is needed, or none if $prefix/include is already in the compiler's
include file search path.

OTOH, nearly all users of the Octave libraries are building extensions
for Octave, not standalone applications, so they typically use the
mkoctfile script that works like a compiler driver so they never have
to deal with the options directly.

For Octave, I don't see the #ifdef method as an option because there
are many include files (nearly 550 are installed for 2.1.73; more for
the current development version) and things are still changing fairly
rapidly so it would be tough to maintain.  But sundials, cvode, cvodes,
ida, and kinsol only have about 10 headers each, so I don't think
dealing with the extra maintenance hassle would be that great (though
of course I am not the one dealing with the maintenance either).

jwe
