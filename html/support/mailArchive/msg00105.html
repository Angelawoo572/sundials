Hi Alan,

Beware of such generalizations :-)
As noted in the documentation, each nonlinear iteration method is better suited for different types of problems: functional iteration for nonstiff problems, Newton iteration for stiff problems. It is therefore very easy to come up with problems for which one will perform much better than the other.

Consider the ODE y'=lambda*y (the famous Dahlquist test problem) with some large negative lambda and see for yourself what happens for the two different nonlinear solvers (functional vs. Newton) if you take a backward Euler step. That should give you a hint on how to construct a simple example for which functional iteration will perform much better than Newton :-)

What was the ODE you tested?

Hi,

I eventually got around incorporating CVODE in my Delphi application. It's working very well. Basically my approach has been as follows:

- Compile the original C code using the free Borland C++ 5.5 compiler (see
http://www.borland.com/downloads/download_cbuilder.html)
- Link the object files into a Delphi unit that gives me access to all the procedures/functions I am interested in, as well to some structures (in particular N_Vector).

That works like a gem, even though I don't have all of the CVODE functionalities. Indeed, at this stage, I can 'only' use the Adams-Moulton and BDF solvers with a functional iterator. Still, it works fine and doesn't require any DLL or anything.

On a side note, I remember reading on this mailing list something about Herbert Sauro (Jarnac) saying that LSODA was much slower than CVODE. I would, in fact, expect LSODA to be better, since it can switch between Adams-Moulton and BDF, depending on the stifness of the problem, right?

Still, besides incorporating CVODE in my application, I also incorporated Herbert's Delphi version of LSODA and like him, I noticed that LSODA was much slower (sometimes even 10 times slower, depending on the problem I am solving). I don't think LSODA itself is to be blamed, but maybe its implementation in Delphi and, more probably in fact, Delphi itself. Indeed, the CVODE version that I have has been compiled in an optimised way (or as optimised as it can be using the Borland C++ 5.5 compiler and as far as I can tell), while Delphi is, unfortunately, known for being rather bad at optimising code. That, I believe, could be the reason as to why the Delphi version of LSODA compares rather unfavourably against CVODE.

I will (in a month or so, as I don't think I will have time before then) try to use a C version of LSODA that I have found on the internet (http://server.ccl.net/cca/software/SOURCES/C/kinetics2/lsoda-dir/).
Incidently, has anyone tried to use this version? Or, does anyone know of a good C version of LSODA?

Cheers, Alan.

Hi Alan and Andreas,

Up to now it is not possible to compile C or C++ Code with Delphi because there isn't a C++/C-Compiler in the package.
This will come with the next major version of Delphi (begin next year).
But Delphi and Borland C++ compilers using the same object format (omf).
Therefore it should be possible to link precompiled object code together.
You have to look carefully to your functions because Delphi and C++ have different calling conventions (see in help for modifier pascal and cdecl).
