Hi André,

Assuming you asked "what do you think?" and not "what do you mean?" :-), I completely agree with you. What you outlined is basically my suggestion (#3 in my previous post) with headers such as ida.h moved to their corresponding directory. I'm fine with this later modification and, moreover, I would probably export all nvector header files (currently only 2 of them - nvector_serial.h and nvector_parallel.h) under ${prefix}/include/nvector/.

The users would then use in their makefiles:

     SUNDIALS_HOME   = <prefix>
     CPPFLAGS        = -I$(SUNDIALS_HOME)/include
     LDFLAGS         = -L$(SUNDIALS_HOME)/lib

and in their code

#include <nvector/nvector_serial.h>
#include <ida/ida.h>
#include <ida/ida_dense.h>
#include <sundials/sundials_types.h>

The only inconvenience of this approach (as John pointed out) is that it uses 
several directories under ${prefix}/include.

--Radu

P.S. This whole discussion picked my interest and I did quite a bit of searching on best/common/standard practices on this issue. What I found (correct me if I'm wrong) is that there really is no standard way of doing things - nothing relevant in the "GNU Coding Standards" for example.

The conditional include that John suggested is something I considered even before the latest SUNDIALS release (the first package where I've seen this done was 'xine' I think) and dismissed it as not very elegant (particularly for a package such as SUNDIALS for which I expect the users to look into the header files - more than a user of xine would do). This is why, in my first message, I asked for solutions that did *not* use this (or some convoluted structure of the source tree, or the use of symbolic links, etc).

Qt, which is often used as an example of good practice, also exports headers 
into several directories under ${prefix}/include.

I also looked at my installation of octave (I have an older version, 2.1.65) and its header files are installed under a directory ${prefix}/include/octave-2.1.65/octave. This means that even if ${prefix}/include is in the compiler's default search path, an example using #include <octave/whatever.h> still needs a -I flag. John, correct me if this is not true.
