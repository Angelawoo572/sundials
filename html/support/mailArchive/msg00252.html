I have to report very bad results with using .NET RedirectStandardError in
VB (the mechanism referred to by Andreas Nicolai's).
What I have is that when the spawned process is restarted, it does not get
the messages anymore; I still have to log the bug with MS.

In rreply to Herbert Sauro, I think a good numeric library should have
something more high level then stderr for reporting and logging errors and
warnings.
Let me point you to this link; maybe we can take some ideas from the realm
of business applications:
http://www.codeproject.com/cpp/loggerservice.asp

Named pipes could be the way to get neat interprocess communication; have
the logging service as an external process may be attractive if the
numerical computations are run repeatedly as is the case for on-line
applications. Also interesting for distributed parallel applications.
But it is an overshoot for a standalone serial number-crunching application.

I suggest all errors and warnings are sent using a standard interface:

void reset_error(int ID);
void set_error(int ID, int severity, std::string &msg);
void set_warning(int ID, int severity, std::string &msg);

The ID is the object identifier so that if this is used inside an OO
environment the logger service can log the instance identifier of the class
that originated the error/warning. 
The library developers could provide a default implementation using stderr
and leave it to the user to provide an alternative implementation using
named pipes or ...

Paolo
