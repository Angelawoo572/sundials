Re: [sundials-users] include file directory layout

From Kai Hartmann <Kai.Hartmann@uni-koeln.de>
Date Wed, 21 Jun 2006 16:10:49 +0200 
> P.S. This whole discussion picked my interest and I did quite a bit of
> searching on best/common/standard practices on this issue. What I found
> (correct me if I'm wrong) is that there really is no standard way of
> doing things - nothing relevant in the "GNU Coding Standards" for example.
> 
> The conditional include that John suggested is something I considered
> even before the latest SUNDIALS release (the first package where I've
> seen this done was 'xine' I think) and dismissed it as not very elegant
> (particularly for a package such as SUNDIALS for which I expect the
> users to look into the header files - more than a user of xine would
> do). This is why, in my first message, I asked for solutions that did
> *not* use this (or some convoluted structure of the source tree, or the
> use of symbolic links, etc).
> 
> Qt, which is often used as an example of good practice, also exports
> headers into several directories under ${prefix}/include.
> 
> I also looked at my installation of octave (I have an older version,
> 2.1.65) and its header files are installed under a directory
> ${prefix}/include/octave-2.1.65/octave. This means that even if
> ${prefix}/include is in the compiler's default search path, an example
> using #include <octave/whatever.h> still needs a -I flag. John, correct
> me if this is not true.
> 

Though I have followed the discussion casually only, I would like to
second the change - and would prefer the headers in one directory. It is
true, you can find many ways of code organization, but I think most of
them have evolved historically and not out of common sense.

The way octave seems to deal with the header files has certain
advantages - you can keep different versions installed and a symbolic
link switches the default directory (ln -s
${prefix}/include/octave-2.1.65/octave ${prefix}/include/octave) or you
can link against a specific version (with -I flag). I think boost is
doing it in the same way.

Using more than one subdirectory (the last proposal) would also be ok,
but I don't see the advantage over the more fine-grained and IMHO better
organized method of putting all these directories into one directory
named sundials. That would be my favorite.

Just for the sake of completeness: no preprocessor conditionals if it
can be achieved in another good way. But this option has left the race
already, I think.

Kai
