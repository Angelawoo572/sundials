On 19-Jun-2006, Radu Serban wrote:

| What should this structure be for options #1 and #2 which you seem
| to favor? Sugestions?

For option 1, SUNDIALS users would use

  #include <sundials/whatever.h>

and -I$prefix/include (and the -I option is only needed if
$prefix/include is not in the compiler's default search path for
include files.   In your sources, you can keep the directory structure
however you like and use

  #if defined (SUNDIALS_LIBRARY)
  #include "other.h"
  #else
  #include <sundials/other.h>
  #endif

When building the SUNDIALS library, you need to define
SUNDIALS_LIBRARY and add whatever -I flags are needed for the source
directory layout you've selected.  Users don't define SUNDIALS_LIBRARY
when compiling their code and they need at most one -I option.  The
same kind of thing will also work for option 2.  This is the method
used by the readline library, for example.

| (3) Header files in a directory structure under ${prefix}/include
| (In this case, the subdirectory containing the headers sundials_***.h would 
be named 'sundials')
| 
| Structure of ${prefix}/include would then be:
| 
| ${prefix}/include
| ${prefix}/include/ida.h
| ${prefix}/include/ida/ida_dense.h
| ${prefix}/include/sundials/sundials_types.h
| 
| A user code would have:
| 
| #include <ida.h>
| #include <ida/ida_dense.h>
| #include <sundials/sundials_types.h>

I think there are a couple of disadvantages to this layout.  First,
you are grabbing several directories of /usr/include (or /usr/local or
/opt) and that is more likely to lead to conflicts with other packages
in the future.  It also means that users have to know more about the
structure of the files.  Why is ida.h at a different level from
ida_dense.h?  FWIW, the readline library also installs all headers
into one directory so you include readline/readline.h and
readline/history.h, not readline/history/history.h or
history/history.h.

jwe
