On 15-Sep-2006, Radu Serban wrote:

| Hi John,
| 
| Indeed, this is a bug in CVodeF. The wrong behavior you see can happen only 
when 
| output is requested more than once within the same internal step (notice that 
| the step size used by cvodes when this error occurs is about 0.14, larger 
than 
| 0.1). The reason for this is that, whether you call CVodeF in NORMAL or 
ONE_STEP 
| mode, it will in turn call CVode always in ONE_STEP mode and the current 
| implementation will take a new step even if not necessary...
| I fixed this in the current development version. If you want to fix your 
CVODES 
| copy, in the file cvodea.c, function CVodeF(), replace the block
| 
|    /* On the first step, load dt_mem[0] */
|    if ( nst == 0) {
|      dt_mem[0]->t = ca_mem->ck_mem->ck_t0;
|      storePnt(cv_mem, dt_mem[0]);
|    }
| 
| starting at line 354 (in Revision: 1.37.2.7), with the block:
| 
|    /* On the first step, load dt_mem[0].
|       On subsequent steps, test if taking a new step is necessary. */
|    if ( nst == 0) {
| 
|      dt_mem[0]->t = ca_mem->ck_mem->ck_t0;
|      storePnt(cv_mem, dt_mem[0]);
| 
|    } else if ( (tn - tout)*h >= ZERO ) {
| 
|      /* If tout was passed, return interpolated solution.
|         No changes to ck_mem or dt_mem are needed. */
|      *tret = tout;
|      flag = CVodeGetDky(cv_mem, tout, 0, yout);
|      *ncheckPtr = nckpnts;
|      newData = TRUE;
|      ckpntData = ca_mem->ck_mem;
|      np = nst % nsteps + 1;
|      return(flag);
| 
|    }
| 
| This should fix the problem.

Yes, it seems to work for me.  Thanks for the quick fix.

jwe
