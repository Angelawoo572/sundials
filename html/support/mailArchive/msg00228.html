Andreas,

Unfortunately, unlike IDA, the user-supplied functions in CVODE/CVODES
do not have a return flag. This is something I've wanted to add for a
while and I may still try to do before the next release. With that, a
RhsFn would return 0 if successful, a positive value if a recoverable
error occured (in which case the solver would try to correct and retry),
and a negative value if an unrecoverable error occured (in which case
the solver would halt).

In the meantime you could try to either:
1) use CVodeSetMaxStep to specify a maximum length of the step size (if
you can estimate such a value)
or
2) use IDA which has the feature mentioned above.

--Radu

> Hi there,

> here's a problem that I sometimes encounter in CVODE simulations. 
> It happens with fine grids where the gradients between elements 
> can become huge (and also the calculated divergences). When the 
> solver then uses a too large time step, I get overflow/underflow 
> of my floating point numbers and the whole simulation virtually 
> crashes. Unfortunately this effect happens even though I limit all 
> values that I get in the y-vector to the allowed ranges.
> 
> Essentially this happens when the solver does an extrapolation 
> with a too large time step (typically when there wasn't anything 
> happing in the solution for a while and suddendly at the boundary 
> there is a large change). Theoretically this should work out fine 
> because CVODE will get (sometimes multiple) convergence failures 
> and reduces the time step accordingly. However, the _first_ 
> extrapolation step will already lead to excessively large 
> divergences (sometimes 6 orders of magnitude higher then typical) 
> before the solver can react to the problem.
> 
> Right now I solve the problem by limiting my values in the ydot 
> vector. However, this is mathematically and physically incorrect.
> 
> Now here's a possible (and better) solution: I could detect these 
> excessive divergences and somehow _tell_ the solver that there 
> will be a convergences problem. The solver should then try to 
> start again with a much smaller time step.  If I assume that these 
> overshots/undershots only happen during the normal iteration and 
> not during the finite-difference-jacoby generation, this would 
> probably work fine.
> 
> 
> Is there already a feature in the CVODE that allows that or is it 
> possible to add such a feature? Basically a function that I could 
> use to set a "failure" flag in the solver?
>
